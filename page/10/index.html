IOC

在.NET MVC中，IoC（Inversion of Control，控制反转）是一种设计原则和实现模式，用于降低代码之间的耦合度，提高可维护性和可测试性。IoC的核心思想是将对象的创建和生命周期管理交由外部容器或框架来控制，而不是由对象自身负责。

在.NET MVC中，IoC容器的使用主要有以下几个目的：

依赖注入（Dependency Injection）：通过IoC容器，可以将对象之间的依赖关系进行解耦。在MVC应用程序中，控制器依赖于服务层或其他组件，这些依赖关系可以通过IoC容器在运行时注入到控制器中，而不是硬编码在控制器中。这样，如果需要替换服务层或组件的实现，只需要修改IoC容器的配置，而不需要修改控制器代码。
生命周期管理：IoC容器负责管理对象的生命周期，包括创建、销毁对象等。在.NET MVC中，IoC容器可以用来创建和销毁控制器实例，以及依赖注入的实例。通过IoC容器，可以统一管理对象的生命周期，避免出现内存泄漏等问题。

单元测试：使用IoC容器可以方便地进行单元测试。在单元测试中，可以通过IoC容器模拟和控制依赖项，使得测试更加简单和可靠。
常见的.NET IoC容器有Unity、Autofac、Ninject等。这些IoC容器提供了丰富的功能和配置选项，可以根据项目的需求选择合适的容器，并通过配置文件或代码来定义和管理依赖关系。

1.DefaultControllerFactory

DefaultControllerFactory是ASP.NET MVC中的一个类，用于创建控制器实例。它是MVC应用程序中控制器工厂的默认实现，当没有显式指定其他控制器工厂时使用。

DefaultControllerFactory通过反射机制来创建控制器实例。它根据传入的控制器名称或类型，通过反射找到对应的类，并调用默认的无参构造函数来创建实例。因此，使用DefaultControllerFactory的控制器必须具有无参构造函数。

然而，在某些情况下，我们可能需要创建具有参数的控制器实例，或者需要根据配置来动态创建控制器实例。这时，我们可以自定义一个控制器工厂类，并覆盖CreateController方法来实现自定义的控制器创建逻辑。

MVC的四种传值方式

  • ViewData:

    ViewData 是一个字典对象，它允许你将数据从控制器传递到视图。
    它通常用于传递简单的数据，如字符串、数字等。
    在控制器中，你可以这样设置 ViewData: ViewData[“Key”] = value;
    在视图中，你可以这样获取数据: var value = ViewData[“Key”];

  • ViewBag:

    ViewBag 是一个动态类型对象，它允许你从控制器向视图传递数据。
    它提供了一种更加动态的方式来设置数据，这意味着你可以在运行时决定属性的名称和值。
    在控制器中，你可以这样设置 ViewBag: ViewBag.Key = value;
    在视图中，你可以这样获取数据: var value = ViewBag.Key;

  • Model:

    Model 是视图的核心概念之一，它允许你传递一个对象或对象的集合到视图。
    这是最常见的方法来传递复杂的数据结构，如实体、DTOs、ViewModel等。
    在控制器中，你可以这样设置 Model: return View(myModel);
    在视图中，你可以这样使用模型: @model MyModelType 和通过模型属性访问数据。

  • TempData:

    TempData 是一个短暂的存储机制，它允许你在同一个请求的后续阶段（例如从控制器到同一个或另一个视图）传递数据。
    它使用 session 机制来存储数据，但当数据被读取后会自动从 session 中删除。
    这使得 TempData 非常适合用于传递一些需要在两个或更多视图之间共享的数据。
    使用方法：在控制器中设置 TempData 属性并传递值，然后在视图中读取它。

PartialPage局部页

“PartialPage” 通常指的是一个网页的部分内容，它可以独立于主页面进行加载、修改或更新。在许多Web开发框架中，例如ASP.NET MVC，partial视图（或partial页面）是一个常见的概念。

使用partial页面有几个好处：

性能优化：如果一个页面的某一部分经常改变，而其他部分保持不变，那么只更新这部分内容而不是整个页面可以大大提高性能。
代码重用：如果多个页面有相同的内容或布局，使用partial页面可以避免重复代码。
更易于维护：如果页面的一部分需要更改，只需更改一个地方，而不是多个地方。

在ASP.NET MVC中，partial视图通常使用@Html.Partial()或@Html.RenderPartial()方法来渲染。

c# 多线程的几种方式

1.什么是线程？

进程作为操作系统执行程序的基本单位，拥有应用程序的资源，进程包含线程，进程的资源被线程共享，线程不拥有资源。

Thread类：这是.NET Framework中最基本的线程创建方式。你可以直接实例化一个Thread对象，并传递一个ThreadStart或ParameterizedThreadStart委托给它，这个委托指向了新线程应该执行的代码。Thread类提供了丰富的属性和方法来控制线程的行为，如启动、暂停、恢复、停止线程，获取线程状态等。然而，由于它提供的底层控制较多，直接使用Thread类进行多线程编程可能会相对复杂。

ThreadPool类：ThreadPool（线程池）是一个在后台运行的线程集合，它可以用来执行任务队列中的任务。当你需要执行一个简短的任务时，可以将这个任务添加到线程池中，而不需要自己创建和管理线程。线程池会自动为你管理线程的创建、销毁和调度。这种方式适用于大量短生命周期的异步操作。

Task和Task类：Task类是在.NET Framework 4.0中引入的，用于表示一个异步操作。与Thread和ThreadPool相比，Task提供了更高级的抽象，使得并发和异步编程更加简单。你可以使用Task.Run或Task.Factory.StartNew方法来启动一个新的异步任务。Task是Task的一个泛型版本，它允许你异步地返回一个结果。

Parallel类：Parallel类提供了并行执行循环和数据转换的方法，可以充分利用多核处理器的性能。例如，你可以使用Parallel.For或Parallel.ForEach方法来并行执行循环体中的代码。这些方法会自动将工作分布到多个线程上，并在所有线程完成后返回。

async和await关键字：从C# 5.0开始，你可以使用async和await关键字来编写异步代码，而不需要直接处理线程或任务。这种方式使得异步编程更加直观和简洁。你可以在方法签名中使用async关键字来表示该方法是异步的，然后在方法内部使用await关键字来等待一个异步操作的完成。编译器会自动为你生成必要的代码来管理异步操作的执行和等待。

2.前台线程和后台线程的区别？

程序关闭时，后台线程直接关闭，但前台线程会执行完后关闭。

通过Thread类新建线程默认为前台线程。其他方式创建的都是后台线程。

14、编程实现

简介

编程思想就是一种解决现实问题的一种思考方式，而每一种语言都是对现实问题描述进行抽象，从而让计算机去解决。

  • 目前主流的四大的编程思想：

POP：面向过程编程(Process-oriented programming )
OOP：面向对象编程(Object Oriented Programming)
SOA：面向服务的架构(Service-Oriented Architecture)
AOP：面向切面编程(Aspect Oriented Programming)

POP：

“面向过程”(Procedure Oriented)是一种以过程为中心的编程思想。
这些都是以什么正在发生为主要目标进行编程，不同于面向对象的是谁在受影响。与面向对象明显的不同就是封装、继承、类，并简写为POP。

面向过程注重过程的。当解决一个问题的时候，面向过程会把事情拆分成： 一个个函数和数据（用于方法的参数） 。然后按照一定的顺序，执行完这些方法（每个方法看作一个个过程），等方法执行完了，事情就搞定了。

  • 优点：性能比面向对象高，因为类调用时需要实例化，开销比较大，比较消耗资源;比如单片机、嵌入式开发、 Linux/Unix等一般采用面向过程开发，性能是最重要的因素。

  • 缺点：没有面向对象易维护、易复用、易扩展

OOP:

面向对象（Object Oriented，OO）作为一种新型的程序设计方法，其是以对象模型为基础进行的抽象过程，并在应用过程中形成了描述自己的抽象概念定义，包括对象、类、封装、继承以及多态等。

面向对象的三个基本特征是：封装、继承、多态。

  • 封装：
    封装最好理解了。封装是面向对象的特征之一，是对象和类概念的主要特性。
    封装，也就是把客观事物封装成抽象的类，并且类可以把自己的数据和方法只让可信的类或者对象操作，对不可信的进行信息隐藏。

  • 继承：
    继承是指这样一种能力：它可以使用现有类的所有功能，并在无需重新编写原来的类的情况下对这些功能进行扩展。

通过继承创建的新类称为“子类”或“派生类”。被继承的类称为“基类”、“父类”或“超类”。
继承的过程，就是从一般到特殊的过程。
要实现继承，可以通过“继承”（Inheritance）和“组合”（Composition）来实现。
在某些 OOP 语言中，一个子类可以继承多个基类。但是一般情况下，一个子类只能有一个基类，要实现多重继承，可以通过多级继承来实现。

继承概念的实现方式有三类：实现继承、接口继承和可视继承。
Ø 实现继承是指使用基类的属性和方法而无需额外编码的能力；
Ø 接口继承是指仅使用属性和方法的名称、但是子类必须提供实现的能力；
Ø 可视继承是指子窗体（类）使用基窗体（类）的外观和实现代码的能力。

在考虑使用继承时，有一点需要注意，那就是两个类之间的关系应该是“属于”关系。例如，Employee 是一个人，Manager 也是一个人，因此这两个类都可以继承 Person 类。但是 Leg 类却不能继承 Person 类，因为腿并不是一个人。
抽象类仅定义将由子类创建的一般属性和方法，创建抽象类时，请使用关键字 Interface 而不是 Class。
OO开发范式大致为：划分对象→抽象类→将类组织成为层次化结构(继承和合成) →用类与实例进行设计和实现几个阶段。

  • 多态:
    多态性（polymorphisn）是允许你将父对象设置成为和一个或更多的他的子对象相等的技术，赋值之后，父对象就可以根据当前赋值给它的子对象的特性以不同的方式运作。简单的说，就是一句话：允许将子类类型的指针赋值给父类类型的指针。

实现多态，有二种方式，覆盖，重载。
覆盖，是指子类重新定义父类的虚函数的做法。
重载，是指允许存在多个同名函数，而这些函数的参数表不同（或许参数个数不同，或许参数类型不同，或许两者都不同）。

我们知道，封装可以隐藏实现细节，使得代码模块化；继承可以扩展已存在的代码模块（类）；它们的目的都是为了——代码重用。而多态则是为了实现另一个目的——接口重用！多态的作用，就是为了类在继承和派生的时候，保证使用“家谱”中任一类的实例的某一属性时的正确调用。

优点：

（1）易学好用。

（2）稳定性好。

（3）可重用性。

（4）维护性好。

缺点：

（1）运行效率较低。

（2）类库庞大。

（3）类库可靠性。

SOA:

面向服务架构，它可以根据需求通过网络对松散耦合的粗粒度应用组件进行分布式部署、组合和使用。服务层是SOA的基础，可以直接被应用调用，从而有效控制系统中与软件代理交互的人为依赖性。

SOA是一种粗粒度、松耦合服务架构，服务之间通过简单、精确定义接口进行通讯，不涉及底层编程接口和通讯模型。SOA可以看作是B/S模型、XML（标准通用标记语言的子集）/Web Service技术之后的自然延伸。

SOA将能够帮助软件工程师们站在一个新的高度理解企业级架构中的各种组件的开发、部署形式，它将帮助企业系统架构者以更迅速、更可靠、更具重用性架构整个业务系统。较之以往，以SOA架构的系统能够更加从容地面对业务的急剧变化。

Soa系统是一种企业通用性架构。

适用场景与具体细节参考以下链接：

优点：

  1. 简单化系统的开发
    由于soa具有组合性，可以利用现有的SOA资源，根据同样的开放标准，在不受平台限制的基础上，可以直接利用现有的资源进行组合，让后在按照自己的客户需求，进行进一步的开放。

  2. 面向企业商业流程
    SOA是基于服务的构造，所以开放的出发点，就是如何解决企业流程中出现的问题。

  3. 更好的适应性和扩展性
    由于soa的组件性，和优良的扩展性以及其组件性等待特征，SOA可以更具不同的需求，进行重新的组合和构造。

  4. 互用性

  5. 对系统的升级，分布，和维护有个更多的优化

  6. 简化了提供，寻找和使用服务的过程

  7. 通过共同资源的利用，减少了开支

缺点：

  1. 减低了系统的性能
  2. 在向标准化过度的转换过程，增加了简介费用
  3. 很多没有太多意义的文件型信息
  4. 对商业流程的计划要求甚高

AOP:

面向切面编程，通过预编译方式和运行期动态代理实现程序功能的统一维护的一种技术。AOP是OOP的延续，是软件开发中的一个热点，也是Spring框架中的一个重要内容，是函数式编程的一种衍生范型。利用AOP可以对业务逻辑的各个部分进行隔离，从而使得业务逻辑各部分之间的耦合度降低，提高程序的可重用性，同时提高了开发的效率。

C# 事件（Event）

事件（Event） 基本上说是一个用户操作，如按键、点击、鼠标移动等等，或者是一些提示信息，如系统生成的通知。应用程序需要在事件发生时响应事件。例如，中断。

C# 中使用事件机制实现线程间的通信。

通过事件使用委托

事件在类中声明且生成，且通过使用同一个类或其他类中的委托与事件处理程序关联。包含事件的类用于发布事件。这被称为 发布器（publisher） 类。其他接受该事件的类被称为 订阅器（subscriber） 类。事件使用 发布-订阅（publisher-subscriber） 模型。

发布器（publisher） 是一个包含事件和委托定义的对象。事件和委托之间的联系也定义在这个对象中。发布器（publisher）类的对象调用这个事件，并通知其他的对象。

订阅器（subscriber） 是一个接受事件并提供事件处理程序的对象。在发布器（publisher）类中的委托调用订阅器（subscriber）类中的方法（事件处理程序）。
